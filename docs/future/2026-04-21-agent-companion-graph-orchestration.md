# 桌宠 Agent 与自动编排图设想

## 背景

GraphiteUI 现在正在逐步把图编辑能力收敛到稳定的数据结构和交互基建上：节点、端口、顺序流、数据流、运行反馈、模板与保存逻辑都在向可程序化操作靠拢。等这些基础设施稳定以后，Agent 不需要通过模拟鼠标点击来“猜测界面”，而是可以通过结构化 JSON 和明确的编辑命令来生成、修改、验证和运行图。

这个设想的核心是：把 Agent 从一个隐藏在后台的能力，变成 GraphiteUI 里一个可见、可交互、可协作的桌宠形象。它既是用户理解系统状态的入口，也是未来自动编排图的交互入口。

## 核心概念

桌宠 Agent 是一个漂浮在 GraphiteUI 画布或侧栏附近的小型角色。它不是简单的装饰，而是一个低干扰的协作入口：

- 用户可以点击它发起对话、请求解释当前图、请求优化节点布局、请求生成新节点或规划整张图。
- 它可以根据当前图状态、用户最近操作、运行错误、校验结果主动给出建议。
- 它可以在用户授权后调用 GraphiteUI 的编辑命令，操作画布、创建节点、连接边、打开面板、定位错误节点。
- 它可以通过动画状态表达当前工作状态，例如待机、观察、思考、建议、执行、等待确认、运行中、完成、失败。

桌宠的重点不是“拟人化替代 UI”，而是让 Agent 的能力有一个稳定、可见、可控的入口，降低用户理解自动化行为的成本。

## 视觉与前端绘制方向

优先考虑不依赖重型图片素材的实现方式，让桌宠能跟随 GraphiteUI 的暖色主题变化，也便于后续做状态动画。

### 方案一：字符与 CSS 绘制

使用 HTML、CSS、少量字符或符号来组合桌宠形象。例如用字符眼睛、表情、简单几何体和 CSS 圆角块构成角色。优点是轻量、可主题化、容易做响应式，也符合 GraphiteUI 当前偏纸张、手工感、低噪声的视觉方向。

适合第一阶段落地：先做一个小而克制的角色，用 CSS animation 实现呼吸、眨眼、点头、思考抖动、提示气泡等状态。

### 方案二：SVG 组件绘制

用 Vue 组件直接渲染 SVG 桌宠。每个身体部件、眼睛、手、表情都可以通过状态切换。优点是视觉更稳定，能做更细腻的动作，也能保持矢量清晰。

适合第二阶段：当桌宠已经证明有价值后，把基础形象升级为可维护的 SVG 状态机。

### 方案三：Canvas 或 Web Animations 增强

Canvas 不适合作为第一选择，但可以用于少量效果，例如思考时的小粒子、运行完成时的轻提示、错误状态的短暂震动路径。核心形象仍建议用 HTML/CSS 或 SVG，避免 Canvas 让交互命中、可访问性和主题适配变复杂。

## 交互行为

桌宠默认保持低存在感，不遮挡节点、端口和线条。它可以停靠在右下角、右侧栏边缘、或画布空白处。用户可以拖动它的位置，也可以折叠成一个小胶囊。

基础状态包括：

- 待机：轻微呼吸、偶尔眨眼，不主动打扰。
- 观察：用户选中节点、拖动连线、打开面板时，桌宠进入短暂观察状态。
- 建议：当检测到校验错误、未连接端口、循环上限异常、输出节点未绑定等情况时，显示短气泡。
- 思考：用户请求规划图或解释图时，角色进入思考动画。
- 执行：Agent 正在创建节点、连线或调整图时，显示执行状态，并在画布上配合高亮变更区域。
- 等待确认：需要用户批准修改、删除、运行或调用外部模型时，桌宠显示确认气泡。
- 完成/失败：任务结束后给出简短反馈，并提供查看详情入口。

主动问候应当有节制。建议只在用户空闲一段时间后出现，例如 15 分钟无操作后最多提示一次；同一会话内限制次数，避免变成干扰。问候内容不应泛泛而谈，而应结合上下文，例如：

- “这个图现在有 2 个 state 没有被任何节点引用，要不要我帮你检查？”
- “当前输出节点还没有绑定输入 state，运行后可能没有可预览结果。”
- “你刚创建了条件节点，要不要我帮你补 true/false/exhausted 三条路线？”

## 操作 GraphiteUI 的方式

Agent 不应该直接操作 DOM，也不应该模拟点击。正确方式是新增一层明确的编辑命令接口，也就是让 Agent 调用和用户操作同级的 GraphiteUI 能力。

建议抽象为 `GraphCommandBus`：

- `createNode(kind, position, preset?)`
- `updateNode(nodeId, patch)`
- `connectState(sourceNodeId, sourcePort, targetNodeId, targetPort)`
- `connectFlow(sourceNodeId, targetNodeId, route?)`
- `removeEdge(edgeId)`
- `focusNode(nodeId)`
- `openPanel(panelId)`
- `validateGraph()`
- `runGraph()`
- `saveGraph()`

所有命令都应该经过同一套事务系统，支持撤销、重做、审计和失败回滚。Agent 每次修改图都应该形成一条可读的操作记录，例如“创建 agent 节点 answer_helper”“连接 question 到 answer_helper”“将 answer 绑定到 output_answer”。

## 自动编排图能力

当 GraphiteUI 的 JSON 图结构、模板系统、校验器和运行反馈稳定以后，Agent 可以通过结构化 JSON 来绘制图。自动编排图建议分成两种模式。

### 模式一：规划后等待人类确认

这是更安全、更适合作为第一阶段上线的模式。

流程是：

1. 用户描述任务目标，例如“做一个能回答知识库问题，并在不确定时追问用户的流程”。
2. Agent 分析任务，生成节点规划，包括输入节点、Agent 节点、条件节点、输出节点、需要的 state、数据流和顺序流。
3. Agent 生成一份图草案 JSON，并渲染到画布上。
4. GraphiteUI 高亮展示新增节点和连接，桌宠解释每个节点的作用。
5. 用户检查、拖动、编辑或删除节点。
6. 用户自己点击运行图。

这个模式的关键是“Agent 负责起草，人类负责确认和运行”。它适合避免错误执行，也符合可视化编排工具的核心价值：用户能看见、理解和调整流程。

### 模式二：边走边画的自主编排

这是更高级的模式，适合在基础能力成熟以后再做。

流程是：

1. 用户给出目标，并明确授权 Agent 自主编排和运行。
2. Agent 先创建最小可行图，例如输入节点、一个 Agent 节点、输出节点。
3. Agent 根据校验结果、运行反馈和中间结果逐步添加节点。
4. 画布上的节点会不断“生长”：新增节点出现、线条连接、条件分支补齐、输出绑定更新。
5. Agent 每一步都记录为什么要这么做，并在必要时暂停请求确认。
6. 当 Agent 认为图已经能实现目标时，自动保存并运行图。
7. 图运行结束后，桌宠通知用户结果，并提供查看运行详情、继续优化或回退修改的入口。

这个模式的关键是“Agent 自己走一步看一步”。它不只是一次性生成 JSON，而是把图编辑、校验、运行反馈和再次编辑变成一个闭环。

## 安全边界

自动编排图必须有清晰边界，否则用户很难信任它。

- 任何删除节点、删除边、覆盖用户编辑、运行会产生费用或调用外部服务的操作，都需要明确确认。
- Agent 的所有图修改都必须可撤销。
- Agent 不能绕过 GraphiteUI 的校验器。
- Agent 不能直接写入不可见状态，必须通过图文档和命令系统修改。
- 自主模式必须有停止按钮，并能暂停在当前图状态。
- 运行失败时，Agent 应解释失败点，而不是直接继续无限重试。

## 与现有 GraphiteUI 的结合点

这个设想依赖几个已经在建设中的能力：

- 图文档 JSON 能完整表达节点、state、数据线、顺序线、条件分支和运行配置。
- 前端画布能可靠地根据 JSON 渲染图。
- 节点创建、连线、删除、编辑都通过统一逻辑完成，而不是散落在组件内部。
- 校验器能告诉 Agent 当前图缺少什么、哪里不合法。
- 运行时反馈能映射回节点和边，让 Agent 知道失败发生在哪里。
- 模板系统能作为 Agent 生成图时的参考和复用单位。

## 分阶段落地建议

第一阶段只做桌宠入口，不让它修改图。它可以显示待机动画、打开对话面板、解释当前选中节点、提示校验问题。

第二阶段让桌宠具备只读上下文能力。它能读取当前图结构、选中节点、运行状态、校验结果，然后给出建议，但仍不直接改图。

第三阶段开放“规划后等待确认”模式。Agent 可以生成图草案 JSON，渲染到画布上，但用户负责确认和运行。

第四阶段开放受控编辑命令。Agent 可以在用户确认后创建节点、连接边、修改配置，并且每一步都可撤销。

第五阶段再考虑“边走边画”的自主编排。这个阶段必须建立在校验、撤销、运行反馈、权限确认都可靠的基础上。

## 长期目标

长期看，桌宠 Agent 可以成为 GraphiteUI 的“协作层”。用户不需要在空白画布上从零思考每一个节点，而是可以先描述目标，由 Agent 起草结构，再由用户审查和微调。对于复杂任务，Agent 也可以像搭积木一样逐步扩展图，在画布上把自己的思考过程可视化。

最终体验应该是：用户能随时看见 Agent 正在做什么、为什么这么做、下一步准备做什么，并且能随时暂停、修改或接管。GraphiteUI 负责提供可见、可控、可验证的图编辑环境；桌宠 Agent 负责把自动化能力变成一个可理解的协作者。
