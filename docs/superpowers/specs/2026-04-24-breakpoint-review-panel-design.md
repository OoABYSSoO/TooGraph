# 断点暂停右侧面板重构设计说明

日期：2026-04-24

## 1. 背景

当前编辑器在命中 agent 断点后，会在右侧打开 `EditorHumanReviewPanel`。现有面板的问题有三类：

1. 首屏优先级不对  
   面板顶部先展示的是通用标题、暂停状态和说明文案，真正的主动作 `Continue Run` 被放在底部，不符合“先确认，再继续”的操作顺序。
2. 信息噪音过多  
   `Paused Run`、`awaiting_human`、`pause_reason` 这类信息在断点场景中基本是恒定语义，重复展示只会挤占首屏空间。
3. 编辑范围过宽  
   当前面板把运行时 state 全量平铺出来，缺少“这次断点之后到底哪些输入真的需要人补”的筛选，因此看起来像通用 inspector，而不是断点决策面板。

用户已经明确要求：

- 首屏第一层放 `继续运行`
- 不强调暂停原因
- 当前暂停节点不必大面积展示，最多保留一个轻量 `聚焦` 动作
- 需要默认展示“当前全部需要填写”的项，而不是只展示 1-2 项
- “当前需要填写”的判定要覆盖从当前断点到下一个断点之间的所有可能分支
- 不只关注后续 agent，也要关注后续 condition 的判断依赖

## 2. 目标

### 2.1 核心目标

1. 让断点面板首屏变成动作优先的任务面板，而不是通用状态面板
2. 把 `继续运行` 提升为唯一高优先级动作
3. 默认只展开“当前窗口内确实需要人填写”的 state
4. 把其他 state 收进次级折叠区，避免抢首屏注意力
5. 保持现有 `/resume` 语义和运行时协议不变，只重构前端展示和前端分析逻辑

### 2.2 非目标

1. 本次不改断点/暂停的后端状态机
2. 本次不引入新的后端接口
3. 本次不做“最可能路径”推断，条件分支统一按保守覆盖处理
4. 本次不处理条件节点分支未连接这类图校验问题
5. 本次不重做整套 state 编辑器，只重排断点面板并新增缺输入分析

## 3. 用户确认后的产品结论

### 3.1 首屏优先级

断点面板首屏应围绕以下顺序组织：

1. 是否继续运行
2. 当前是否还有必须人工补的输入
3. 需要补哪些输入
4. 其他 state 作为次级信息

因此首屏不再保留：

- 大标题式 `Paused Run`
- `awaiting_human` 状态徽章
- `pause_reason`
- 大块当前节点说明卡

### 3.2 当前节点信息的处理

用户明确表示当前暂停节点不需要大面积展示，因为命中断点后画布已经会自动聚焦。

保留策略：

- 面板顶部只保留一个轻量 `聚焦` 按钮，用于再次回到当前断点节点

去除策略：

- 不再单独保留当前节点标题卡
- 不再在首屏重复描述“当前暂停在谁”

### 3.3 “当前需要填写”的定义

用户确认后的规则如下：

1. 计算窗口从当前断点之后开始
2. 遍历到下一个断点之前；如果后面没有下一个断点，则遍历到流程结束
3. 遇到条件分支时，保守覆盖所有可能分支
4. 在该窗口内，既看后续 agent 的读取，也看后续 condition 的判断读取
5. 如果某个 state 会在该窗口内被 agent 或 condition 使用，但没有明确输入来源，则归入“当前需要填写”

这里的“没有明确输入来源”指：

- 没有 input 节点写入它
- 没有在该读取点之前稳定可达的上游写入

不纳入首屏必填判定的情况：

- output 节点的读取
- 条件节点分支未连接

## 4. 新的面板结构

新的 `EditorHumanReviewPanel` 从上到下拆成四层。

### 4.1 顶部行动条

顶部第一层固定为行动条：

- 左侧：`继续运行` 主按钮
- 右侧：`聚焦` 轻量按钮

规则：

- `继续运行` 是唯一主色按钮
- `聚焦` 为次级按钮或图标按钮
- 不再让任何静态说明占据比 `继续运行` 更高的视觉权重

### 4.2 任务摘要行

行动条下方保留一行极短摘要，用来替代当前的暂停说明卡：

- `到下一个断点前，需人工填写 2 项输入`
- 如果没有必填项：`当前断点后没有需要人工补充的输入`

该摘要只回答一个问题：现在是否需要动手补数据。

### 4.3 当前必填区

默认展开并展示当前窗口内全部需要填写的 state。

每项卡片只保留：

- state 名称
- 一句短描述（如果定义中有说明）
- 输入控件
- 可选的小型 `待填写` 提示

默认不保留这些元素：

- 大块状态标签
- 重复的 run 状态
- 冗余的类型标签
- 与当前任务无关的装饰信息

### 4.4 其他 state 折叠区

所有不属于“当前需要填写”的 state 都移动到折叠区，例如：

- `其他 state (7)`

默认关闭。展开后可以继续查看和编辑，但不应影响首屏任务流。

## 5. 缺输入分析模型

### 5.1 新增前端分析输出

`humanReviewPanelModel.ts` 不再只返回全量 `reviewRows`，而是需要新增一个更高层的面板模型，至少包含：

```ts
type HumanReviewPanelModel = {
  requiredNow: HumanReviewRow[];
  otherRows: HumanReviewRow[];
  requiredCount: number;
  hasBlockingEmptyRequiredField: boolean;
  summaryText: string;
};
```

其中：

- `requiredNow`：当前窗口内需要人工填写的全部项
- `otherRows`：其他仍可查看/编辑的 state
- `requiredCount`：首屏摘要用
- `hasBlockingEmptyRequiredField`：继续运行前的阻塞判定
- `summaryText`：顶部摘要文案

### 5.2 遍历窗口的定义

前端分析逻辑需要基于：

- `document`
- `run.current_node_id`
- 图中的断点配置
- state 读写关系

构造一个从“当前断点之后”开始的前视窗口。

停止条件：

1. 命中下一个断点节点
2. 到达流程末端

条件节点策略：

- 把所有可能分支都纳入遍历
- 不做“最可能分支”裁剪

### 5.3 候选 state 的收集

在遍历窗口内，收集所有会被以下节点消费的 state：

- `agent.reads`
- `condition.reads`

同一个 state 即使在多个节点被消费，也只保留一次，并记录：

- 第一次消费距离
- 是否出现在多个分支
- 首次消费节点类型

### 5.4 明确输入来源的判定

某个候选 state 若满足任一条件，则不进入 `requiredNow`：

1. 图中存在 input 节点写入该 state
2. 在当前窗口中，所有到达该消费点的路径上都有稳定上游写入覆盖该 state

否则进入 `requiredNow`。

这里的判断目标不是证明“运行时一定为空”，而是识别“图结构上没有稳定输入保障，因此应由人类补齐”的项。

### 5.5 排序规则

`requiredNow` 的默认排序规则：

1. 首次消费距离当前断点更近的优先
2. 被更多后续分支共同依赖的优先
3. 最后按 state 名称稳定排序

这样首屏顺序更接近真实执行优先级。

## 6. 继续运行交互

### 6.1 默认行为

`继续运行` 按钮始终固定在最上方，不因为下方内容变多而下沉。

### 6.2 必填未完成时

如果 `requiredNow` 中仍存在明显未填项，点击 `继续运行` 时不直接提交。

此时执行：

1. 顶部展示短提示：`还有 N 项需要填写`
2. 自动滚动到第一项未填内容
3. 对该项做一次轻量高亮

这样既满足“按钮在第一层”，也避免误继续。

### 6.3 无必填项时

如果当前窗口没有需要人工补的数据，则：

- 仍显示 `继续运行`
- 摘要显示“当前断点后没有需要人工补充的输入”
- 可以直接继续，不出现空表单

## 7. 视觉方向

### 7.1 面板气质

断点面板应从“通用 inspector”转成“任务决策面板”。

具体表现：

- 首屏元素更少
- 行动按钮优先于信息卡
- 必填区比次级信息更明确
- `其他 state` 明确收进折叠区

### 7.2 视觉压缩

建议同步做以下减法：

1. 去掉现有 `run-card` 大卡片
2. 去掉重复状态徽章
3. 压缩顶部留白
4. 必填卡片改成更轻的表单行或轻卡片
5. 弱化不必要的边框层级

### 7.3 保留的语义锚点

面板顶部仍保留极少量语义锚点即可：

- 面板归属仍是人类审核/断点继续
- 有轻量 `聚焦` 动作
- 有任务摘要说明当前是否缺输入

## 8. 实现范围

本次建议只改这三块：

1. `frontend/src/editor/workspace/EditorHumanReviewPanel.vue`
2. `frontend/src/editor/workspace/humanReviewPanelModel.ts`
3. 对应的 structure/model tests

不扩散到：

- 后端 runtime
- `/resume` 接口
- 运行详情页
- 普通 state 面板

## 9. 测试策略

至少补齐以下验证：

### 9.1 模型测试

1. 能正确识别当前窗口内的 `requiredNow`
2. 遇到条件分支时按保守策略覆盖所有分支
3. 遇到下一个断点时停止继续向后遍历
4. 有 input 节点或稳定上游写入时，不进入必填区
5. `condition.reads` 也能触发必填判定

### 9.2 结构测试

1. 面板顶部存在 `继续运行`
2. 顶部存在轻量 `聚焦` 按钮
3. 不再渲染暂停原因相关主内容
4. 存在 `其他 state` 折叠区
5. `requiredNow` 区优先于其他 state 展示

### 9.3 交互测试

1. 必填为空时可以直接继续
2. 必填未完成时点击继续会阻止提交并指向第一项未填内容
3. 展开/折叠 `其他 state` 不影响 `requiredNow`

## 10. 风险与取舍

### 10.1 风险

“稳定上游写入”的判定如果做得过于简单，可能把本来无需人工填写的 state 错判为必填。

### 10.2 当前取舍

本次优先接受“略保守”的结果，也就是宁可多提示一项，也不要漏掉断点之后真正会卡住 agent 或 condition 的缺输入。

这与用户选择的“条件分支保守覆盖所有可能分支”是一致的。

## 11. 结论

断点暂停右侧面板应被重新定义为：

- 一个以 `继续运行` 为首屏核心动作的任务面板
- 一个只默认展开“当前断点之后确实需要人工补充”的输入区
- 一个把其他 state 收纳到折叠区的次级查看器

这次改动的本质不是“美化一张卡片”，而是把断点暂停的交互语义从“浏览所有 state”改成“补齐后续必需输入，再继续执行”。
