# 编排器 Tabs 工作区设计

## 背景

当前 `/editor` 已经从历史的“编排器入口页”收口成模板与图的目录页，但它仍然不是一个真正意义上的持续工作区。

现状存在一个核心问题：

- 用户在编排某张图时离开 `/editor`
- 去技能页、设置页、运行页或首页做其他操作
- 再次返回 `/editor` 时，不能稳定回到刚才正在编辑的那组 graph 上下文

这使 `/editor` 更像目录页，而不像实际的编排工作区。

## 目标

把 `/editor` 改造成真正的 tabs 工作区，使其具备以下能力：

1. 支持同时打开多个 graph
2. 顶部以 tab bar 管理当前打开的图
3. 离开 `/editor` 后再次回来，能够恢复上次打开的整组 tabs
4. 当前活跃 tab 与 URL 保持一致
5. 未保存修改的 tab 在关闭时有明确的保存确认

## 核心决策

### 1. `/editor` 是工作区，不再是单纯目录页

`/editor` 的主职责从“展示模板和已有图的目录”变成：

“承接当前编排工作现场”

因此：

- 如果存在已打开 tabs，进入 `/editor` 时应直接恢复工作区
- 只有在没有任何已打开 tabs 时，才显示欢迎态

### 2. 欢迎态只在无 tabs 时出现

当本地没有任何已打开 tab 时，`/editor` 显示欢迎态，而不是模板与图并列的大目录页。

欢迎态应提供三个明确动作：

- `新建图`
- `从模板创建`
- `打开已有图`

这三个动作都是“打开一个 tab”的入口，而不是把用户带离工作区心智。

### 3. 恢复整组 tabs，而不是只恢复最后一个 graph

重新进入 `/editor` 时，应恢复：

- 上次打开的全部 tabs
- tab 顺序
- 最后活跃的 tab

而不是只记住最后一个 graph。

原因：

- 真正的编排工作常常需要在多张图之间切换
- 单图恢复只能解决一部分问题，不能保留完整工作现场

### 4. 同一个 graph 不允许重复打开多个 tab

如果用户再次打开一张已经在 tabs 中存在的 graph：

- 不新建重复 tab
- 直接切换到现有 tab

这样可以避免：

- 同一张图出现多个副本
- dirty 状态混乱
- 用户不知道自己改的是哪个副本

### 5. URL 跟随当前活跃 tab

URL 策略采用以下规则：

- 没有任何 tab：`/editor`
- 当前活跃 tab 是已保存 graph：`/editor/<graphId>`
- 当前活跃 tab 是未保存的新图或刚从模板创建、但还没保存成 graph 的临时 tab：`/editor`

原因：

- 已保存图可以通过 URL 精确定位
- 未保存 tab 没有稳定后端身份，不应把临时 id 暴露到 URL

### 6. 关闭 dirty tab 时必须弹确认框

当用户关闭一个有未保存修改的 tab 时，弹出确认框，并提供三个动作：

- `保存并关闭`
- `不保存，直接关闭`
- `取消`

行为定义：

- 选择 `保存并关闭`：
  - 先执行保存
  - 保存成功后关闭 tab
  - 若关闭后仍有其他 tab，则切换到合适的剩余 tab
  - 若这是最后一个 tab，则回到欢迎态
- 选择 `不保存，直接关闭`：
  - 直接关闭 tab
- 选择 `取消`：
  - 保持原 tab 不变
- 如果保存失败：
  - tab 不关闭
  - 保持留在当前图
  - 展示错误信息

## 工作区模型

### Tab 数据结构

每个 tab 至少应包含以下字段：

- `tabId`
- `kind`
  - `existing`
  - `new`
  - `template`
- `graphId`
  - 对未保存 tab 可以为空
- `title`
- `dirty`
- `templateId`
  - 仅对从模板创建的临时 tab 有意义
- `createdAt`
- `updatedAt`

### 工作区数据

工作区状态至少包含：

- `tabs`
- `activeTabId`

持久化建议：

- 写入 `localStorage`
- 首轮不落数据库

原因：

- 当前需求是恢复本地工作现场
- 没有必要在第一阶段引入用户级远端工作区同步

## 页面结构

### `/editor` 工作区

`/editor` 页面结构应变成：

1. 顶部 tab bar
2. 当前活跃 tab 对应的画布与编辑区

tab bar 行为包括：

- 切换 tab
- 关闭 tab
- 新建 tab

### `/editor` 欢迎态

当没有 tabs 时，展示欢迎态。

欢迎态内容建议为：

- 一句简短说明
- 三个主动作卡片或按钮：
  - `新建图`
  - `从模板创建`
  - `打开已有图`

其中：

- `从模板创建` 可以打开模板选择器
- `打开已有图` 可以打开最近图或图选择器

欢迎态是工作区内部的“空状态”，不是新的 landing page。

## 与现有路由的关系

### `/editor`

- 无 tabs 时：显示欢迎态
- 有 tabs 时：恢复 tabs 工作区

### `/editor/<graphId>`

访问已有 graph URL 时：

- 若该 graph 已在 tabs 中：
  - 直接切换到对应 tab
- 若该 graph 不在 tabs 中：
  - 将其加入 tabs 并激活

### 新建图 / 从模板创建

这两类入口创建的是“未保存 tab”：

- 在保存前，URL 保持 `/editor`
- 保存后，如果拿到正式 `graphId`，再切到 `/editor/<graphId>`

## 首页与编排器的关系

首页继续承担“从哪里进入”的职责：

- 最近运行
- 模板选择
- 最近图

`/editor` 负责“当前正在编辑什么”：

- 当前打开的 tabs
- 当前活跃图
- 当前工作现场恢复

因此首页与 `/editor` 的职责完全不同，不再冲突。

## 非目标

本轮不处理以下内容：

- 多窗口或多浏览器实例之间的 tabs 同步
- 用户级远端工作区同步
- tabs 拖拽排序
- 复杂模板搜索和分类系统
- 浏览器原生多文档撤销栈协调

## 验证标准

1. `/editor` 在无 tabs 时显示欢迎态
2. 用户可以从欢迎态打开新图、模板图或已有图，并形成 tab
3. 返回 `/editor` 时可以恢复上次打开的整组 tabs
4. 活跃的已保存图会同步到 `/editor/<graphId>`
5. 未保存 tab 保持在 `/editor`
6. 已打开 graph 再次打开时不会产生重复 tab，而是直接切换
7. 关闭 dirty tab 时会弹出“保存 / 不保存 / 取消”的确认
8. 保存失败时 tab 不会被关闭
